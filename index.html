Tool-by-tool Analysis

도구별 강점, 약점, 적합한 사용처

Codex CLIHigh

baseline runtime. 로컬 코드 변경, 테스트, git diff, sandbox/approval을 수행하는 중심 실행기.

Best for

  • repo 단위 구현
  • 테스트/리팩토링
  • non-interactive exec

Avoid when

  • 장기 스케줄링이 핵심
  • 사용자 memory/gateway가 필요
LazyCodex / OmOHigh local

Codex lifecycle에 hooks/skills/MCP를 연결해 planning과 verified completion을 강화한다.

Best for

  • Codex 안 evidence loop
  • ultrawork discipline
  • plugin hook 기반 강화

Caveat

  • doctor warning 존재
  • composer command surface 의존
Gajae-CodeMedium

독립 runner로 계획·비평·실행·증거를 명시하고 tmux/worktree/team으로 병렬/장기 작업을 다룬다.

Best for

  • 승인 가능한 계획
  • tmux visible lanes
  • coordinator bridge

Caveat

  • 로컬 gjc 미설치
  • beta stage
Hermes AgentHigh

메신저, 스케줄, memory, skills, MCP, subagent를 묶는 지속형 coordinator.

Best for

  • Discord/Telegram 운영
  • 반복 작업/cron
  • 리서치와 조율

Caveat

  • coding runtime 자체보다 조율자 성격
  • 모델/도구 설정 품질 의존
Suggested Workflow

실전 운영 플로우

1

Research / Planning

Hermes에서 리서치와 일정/채널 조율을 하고, 필요한 경우 Gajae의 deep-interview/ralplan이나 LazyCodex ulw-plan 성격의 계획 단계를 사용한다.

2

Implementation

코드 구현은 Codex를 baseline으로 두고, Codex 내부 discipline이 필요하면 LazyCodex/OmO를 켠 상태에서 진행한다.

3

Long-running / Parallel

작업이 길고 눈으로 추적해야 하면 Gajae-Code의 tmux/worktree/team lane을 후보로 둔다. 여러 독립 workstream은 Hermes delegate_task로 분리할 수 있다.

4

Evidence / Report

테스트 출력, git status, doctor warning, public docs/source index를 분리해 GitHub Pages 리포트에 남긴다.

Final Recommendation

지금 당장 추천 조합

기본 조합

Hermes → Codex + LazyCodex. Hermes가 사용자 채널/기억/스케줄을 잡고, Codex+LazyCodex가 repo 내부 구현과 evidence loop를 담당한다.

리스크 큰 작업

계획 승인 gate를 별도로 둔다. Gajae-Code의 plan-before-mutation 철학 또는 Hermes subagent two-stage review로 spec review와 quality review를 분리한다.

다음 실험

gjc를 로컬에 설치/스모크 테스트한 뒤, Hermes coordinator bridge와 실제 tmux/worktree lane 연결을 검증한다.
