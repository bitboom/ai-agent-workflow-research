Executive Summary

핵심 요약

01

모델보다 control plane을 본다

coding agent 품질은 LLM 이름만으로 결정되지 않는다. context를 어떻게 만들고, 어떤 tool을 허용하며, 실패를 어떻게 검증·기록하는지가 핵심이다.

02

Agent loop의 위치가 제품 성격을 가른다

IDE extension, CLI, benchmark runner, persistent coordinator는 같은 “코딩 에이전트”라도 loop와 state의 위치가 다르다.

03

Context assembly가 숨은 병목이다

repo map, active file, search result, memory, failing test log를 고르는 방식이 patch 품질과 비용을 좌우한다.

04

Verification은 결과와 과정 모두를 봐야 한다

테스트 통과만으로 충분하지 않다. agent가 어떤 파일을 읽고 어떤 tool을 썼는지 trajectory까지 기록해야 한다.

05

Gajae/Hermes/OmO는 control-plane 연구 대상이다

Gajae는 `.gjc`, Hermes는 persistent runtime, OmO는 Codex plugin/harness layer로 서로 다른 orchestration 실험을 보여준다.

06

UltraResearch는 초안, controller evidence가 canonical이다

agent 산출물은 가설 생성기로 쓰고, public claim은 source path·runtime trace·artifact·controller cross-check만 승격한다.

Main Question

이 책의 질문

이 프로젝트는 “Codex가 더 나은가, Aider가 더 나은가”를 묻지 않습니다. 대신 coding agent라는 시스템이 under the hood에서 어떤 층으로 구성되는지를 묻습니다.

좋은 coding agent 분석은 기능표가 아니라 실행 루프, 컨텍스트 조립, tool boundary, verification gate, durable state를 추적하는 일이다.

그래서 첫 화면은 링크 모음이 아니라 독자가 순서대로 읽을 수 있는 목차형 입구로 바꿨습니다. 상세 근거는 부록으로 남기되, 본문은 책처럼 한 장씩 이어집니다.

읽는 법

먼저 1–3장을 읽고 공통 구조를 잡은 뒤, 4–6장에서 구현체와 control plane을 비교하고, 마지막으로 evidence appendix에서 claim을 검증합니다.

Taxonomy

Agent는 loop 위치로 나눈다

“AI coding agent”라는 이름은 너무 넓습니다. 분석을 위해서는 agent loop가 어디서 돌고 state가 어디에 남는지부터 나눠야 합니다.

IDE embedded

Cursor, Cline, Continue, Roo. editor index와 diff UX가 강하지만 hidden context와 stale index가 위험하다.

CLI coding agent

OpenAI Codex CLI, Aider, Goose, Claude Code. local shell/test 재현성이 강하지만 sandbox·approval mismatch를 봐야 한다.

Repo-local runner

SWE-agent, OpenHands, Agentless. issue → inspect → edit → test → retry 흐름과 benchmark trace가 핵심이다.

Workflow orchestrator

Gajae-Code, LazyCodex/OmO. planning, review, state, artifact, team lane을 강제하는 control layer다.

Persistent runtime

Hermes처럼 session을 넘어 memory, cron, delegation, gateway, messaging identity로 이어지는 runtime이다.

Team / multi-agent

진짜 multi-agent는 role prompt가 아니라 durable queue, state root, worktree/tmux/mailbox, recovery path가 있어야 한다.

Common Architecture

공통 under-the-hood 구조

각 제품의 UI는 다르지만 반복해서 등장하는 구조는 비슷합니다. 아래 여덟 축을 보면 agent claim을 과장 없이 읽을 수 있습니다.

1

Model client / router

provider, streaming, retry, token budget, task routing을 담당한다.

2

Prompt / context assembly

system rule, repo file, memory, tool output, failing test log를 선택·압축한다.

3

Repo index / code graph

lexical search, AST, LSP, vector index, repo map이 여기에 속한다.

4

Planning / tool loop

plan 생성, action 선택, observation 반영, 반복 중단 조건을 다룬다.

5

Edit / execution

patch, search/replace, whole-file rewrite, shell command, browser/MCP tool을 실행한다.

6

Sandbox / approval

filesystem, network, shell, human approval, enterprise policy를 통제한다.

7

Verification / evidence

test, lint, build, benchmark, reviewer agent, evidence ledger를 남긴다.

8

Memory / durable state

session, checkpoint, skills, long-term memory, workflow artifact를 관리한다.

이 장의 결론

agent를 비교할 때는 “무슨 모델을 쓰는가”보다 “이 여덟 층이 공개·검증·재현 가능한가”를 먼저 본다.

Implementation Atlas

구현체별 읽기 지도

상세 source path는 부록의 atlas에 두고, 본문은 읽는 방향만 남겼습니다.

OpenAI Codex / Aider / Goose

CLI agent의 핵심은 local tool loop, patch 전략, sandbox/approval, git/test feedback이다. Aider의 RepoMap처럼 context compression이 명시된 구현은 연구 가치가 높다.

SWE-agent / OpenHands / Agentless

benchmark runner와 runtime platform 계열은 trajectory, container/runtime, test oracle을 함께 봐야 한다. Agentless는 “항상 autonomous loop가 필요한가”라는 반례도 제공한다.

Cline / Roo / Continue

IDE 계열은 editor state, index freshness, approval UX, remote workspace routing이 핵심이다. 좋은 UX가 hidden context failure를 가릴 수 있다.

Gajae-Code / Hermes / LazyCodex·OmO

이 프로젝트의 현재 deep dive 축이다. Gajae는 `.gjc` artifact replay까지 확인했고, Hermes는 runtime/source drift, OmO는 manifest와 runtime-callable surface를 분리한다.

대표 evidence: source-level architecture atlas에서 SHA-pinned source map을 확인할 수 있다.

Control Plane Triad

Gajae · Hermes · OmO는 따로 읽는다

세 시스템은 coding agent 제품이라기보다 agent workflow를 통제하는 control plane입니다. 따라서 기능표보다 state root와 runtime proof를 봐야 합니다.

Gajae-Code

.gjc/_session-* state, deep-interview → ralplan → ultragoal → team workflow. 현재 `gjc/0.7.3` smoke와 controlled workflow artifact replay가 확인됐다.

Hermes Agent

~/.hermes/ 기반 persistent runtime. local runtime `0.12.0`과 source clone `0.17.0` drift를 분리해 읽어야 한다.

LazyCodex / OmO

Codex 위 plugin/harness layer. manifest-declared hooks/MCP와 실제 runtime-callable `$...` workflow proof를 분리한다.

현재 판정

Gajae는 basic runtime-confirmed + workflow artifact-backed. Hermes와 OmO는 각각 version drift와 runtime-callable surface 검증이 다음 핵심이다.

Evaluation

데모가 아니라 평가 기준을 세운다

작은 toy task 하나를 여러 agent에 던지는 방식은 재미는 있지만, under-the-hood 연구에는 약합니다. 먼저 trace schema와 rubric이 있어야 합니다.

Task success

요구사항이 실제로 충족됐는가. 단순 빌드 성공과 semantic correctness를 분리한다.

Context localization

agent가 필요한 파일·테스트·문서를 찾아 읽었는가.

Tool-use trajectory

위험한 shell, 불필요한 rewrite, 반복 loop, approval bypass가 없었는가.

Verification honesty

실제로 테스트했는가, 실패를 숨기지 않았는가, evidence를 재현 가능하게 남겼는가.

다음 실험 순서: live team/recovery/Hermes bridge trace → benchmark task template → 5–10개 curated task pilot.

Glossary

이 문서의 판정 라벨

source-confirmed 소스 확인
repo clone 또는 package source에서 직접 확인한 claim.
runtime-confirmed 실행 확인
현재 환경에서 command가 exit 0로 동작하고 결과를 반환한 claim.
artifact-backed 산출물 기반
실행 후 `.gjc`, `.omo`, trace, ledger 같은 durable artifact가 남은 claim.
manifest-confirmed 선언 확인
plugin manifest나 docs에 선언되어 있으나 실제 callability는 별도 검증이 필요한 claim.
unverified 미검증
source나 runtime proof 없이 추정 또는 agent draft에만 있는 claim.
