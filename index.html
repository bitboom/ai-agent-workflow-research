Executive Summary

핵심 요약

01

목적은 Workflow Control Plane 분석이다

이 프로젝트는 AI agent 제품 순위표가 아니다. agent를 업무 workflow 안에서 어떻게 phase화하고, 통제하고, 기록하고, 복구하는지 분석한다.

02

AI Agent와 Harness와 Orchestrator를 분리한다

Agent는 tool/edit/test loop, Harness는 task/environment/score, Orchestrator는 workflow/phase/state/artifact/approval/recovery가 중심이다.

03

deep-interview → ralplan → ultragoal → team phase와 .gjc/_session-* state root가 핵심이다.

04

memory, skills, gateway, cron, delegation, MCP/ACP로 session 밖의 agent runtime을 운영한다.

05

Codex 위에 hooks, $... workflows, MCP declarations, .omo artifacts를 얹는 workflow harness/control overlay다.

06

근거 라벨이 본문을 지킨다

UltraResearch와 controller cross-check는 provenance로 남기고, 본문 claim은 source/runtime/artifact/manifest/unverified 라벨로 구분한다.

Main Question

중심 질문

질문은 하나다. Coding Agent는 이제 단일 AI Agent가 아니라, workflow를 설계·통제·기록·복구하는 Workflow Orchestrator로 진화하고 있는가? 그리고 Gajae-Code, Hermes Agent, LazyCodex/OmO는 그 진화를 각각 어떤 방식으로 구현하고 있는가?

좋은 coding agent 분석은 모델 이름이나 기능 수가 아니라, phase model, durable state, artifact ledger, approval gate, worker orchestration, recovery path를 추적하는 일이다.

따라서 Codex, Aider, SWE-agent, OpenHands, IDE 계열은 중요한 배경 비교군으로 남긴다. 본문의 중심은 세 workflow/control-plane 시스템이다.

읽는 법

1–2장에서 목적과 분류 프레임을 잡고, 3–5장에서 Gajae/Hermes/LazyCodex deep dive를 읽은 뒤, 6–7장에서 발전 방향과 남은 검증을 확인한다.

Analysis Frame

AI Agent / Harness / Workflow Orchestrator

AI Agent

하나의 목표를 받아 context를 구성하고 reasoning/tool call/edit/test loop를 돌려 결과물을 만든다. 중심 단위는 turn, message, tool call, patch다.

Harness

agent를 task/environment/benchmark에 넣고 trajectory와 score를 관찰한다. 중심 단위는 task, environment, score, oracle이다.

Workflow Orchestrator

agent/harness를 worker로 삼아 phase, state root, artifact ledger, approval gate, recovery, team coordination을 통제한다.

Control-plane Axes

무엇을 보면 Orchestrator인가

1

Phase model

interview, plan, execute, review, team 같은 workflow 단계가 명시되는가.

2

State root

.gjc, .omo, ~/.hermes처럼 canonical state가 남는가.

3

Artifact ledger

final answer가 아니라 spec, plan, goal, test, review artifact가 남는가.

4

Approval / gate

위험한 실행과 phase 전환을 승인·검증 gate로 통제하는가.

5

Worker orchestration

subagent, tmux worker, background process, cron, external runner를 조율하는가.

6

Recovery / resume

중단 뒤 어디서 이어갈지 state와 artifact로 재구성할 수 있는가.

Background Baselines

일반 agent 비교는 기준선이다

Codex CLI, Aider, SWE-agent, OpenHands, Cline, Roo, Continue 같은 프로젝트는 여전히 필요하다. 다만 이 책에서는 중심 대상이 아니라 기준선이다. 이들은 single-agent loop, benchmark harness, IDE embedded runtime의 baseline을 제공하고, 세 control-plane 시스템이 무엇을 추가하는지 보여주는 배경 비교군이다.

CLI / IDE agent baseline

Codex, Aider, Cline, Roo, Continue는 context assembly, edit loop, sandbox/approval, local verification의 기준선을 만든다.

Harness / benchmark baseline

SWE-agent, OpenHands, Agentless 계열은 task/environment/trajectory/score를 통해 agent를 평가하는 기준선을 제공한다.

읽는 순서: source-level architecture atlas는 Appendix의 background evidence로 유지한다. 본문은 Gajae/Hermes/LazyCodex deep dive를 먼저 읽도록 구성했다.

Three Workflow Control Planes

세 시스템의 분류

Gajae-Code

Repo-local Workflow Orchestrator. deep-interview → ralplan → ultragoal → team phase와 .gjc/_session-* state root가 핵심이다. controlled workflow artifact replay가 강한 근거다.

Hermes Agent

Persistent Agent Runtime / Meta-Orchestrator. memory, skills, gateway, cron, delegation, MCP/ACP로 session 밖 agent runtime을 운영한다.

LazyCodex / OmO

Codex-embedded Workflow Harness / Control Overlay. Codex 위 hooks, $... skills, MCP declarations, .omo artifacts를 얹는다. standalone orchestrator claim은 아직 후보로 둔다.

Evolution

현재 Coding Agent의 발전 방향

Prompt loop → Workflow phase

즉시 구현보다 interview, plan, execute, review 단계가 중요해진다.

Chat transcript → Durable state

대화창이 아니라 state root와 artifact ledger가 canonical source가 된다.

Final answer → Artifact ledger

완료 선언보다 spec, plan, goal, test, review evidence를 남긴다.

Single agent → Worker/process orchestration

subagent, team, cron, tmux, MCP/ACP, external runner가 함께 조율된다.

Glossary

이 문서의 판정 라벨

source-confirmed 소스 확인
repo source, package source, official docs에서 확인한 claim.
runtime-confirmed 실행 확인
현재 환경에서 command가 성공한 claim.

unverified 미검증
아직 source/runtime/artifact proof가 부족한 claim.
